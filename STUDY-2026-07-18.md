### 1.4 为什么只对个人站做替换

判断逻辑在 [src/services/codebuddy/config.js#L172-L181](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/config.js#L172-L181)：

```javascript
const PERSONAL_HOSTS = [LEGACY_CODEBUDDY_PERSONAL_HOST, 'www.codebuddy.ai'];

export function isPersonalHost(host) {
    return PERSONAL_HOSTS.includes(host);
}
```

腾讯 CodeBuddy 后端在不同部署环境下的策略不一样：

| 上游类型 | host 示例 | content_filter 行为 |
|---|---|---|
| 个人站 | `www.codebuddy.ai`、`copilot.tencent.com` | 严格扫描 `claude/anthropic` 等关键词，命中即 `content_filter` |
| 企业/自定义站 | 用户自建/企业租户域名 | 没有这个限制，腾讯不会对租户自部署做这种关键词审查 |

**为什么不在企业站替换：**
- 没必要：企业站不触发 content_filter，替换属于过度处理
- 风险：替换是破坏性的（`anthropic` → `tencent`），企业租户后台若做文本搜索/审计会看到被篡改的字段
- 零收益还可能踩坑

触发分支在 [src/services/codebuddy/api.js#L169-L175](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/api.js#L169-L175)：

```javascript
const host = new URL(getCodebuddyBaseUrl(baseUrl)).host;
if (isPersonalHost(host)) {
    sanitizePayload(requestPayload);
}
```

### 1.5 已知缺口

- `payload.tools` 用 `JSON.stringify` 再 `JSON.parse` 的方式替换，若工具名/描述里含特殊字符可能影响后续解析（已用 try/catch 兜底失败时跳过，但意味着该请求的工具定义完全丢失）
- 只在 `isPersonalHost(host)` 时才替换，自定义/企业上游不替换
- 不替换请求头、URL、metadata 等非 payload 字段

## 3. CodeBuddy 渠道 Credits 计算机制

### 3.1 核心结论

代理**不计算** Credits，只**透传**。CodeBuddy 服务端 SSE 流式响应的最后一个 chunk 的 `usage` 字段里会带一个 `credit` 字段，代理只负责把它**原封不动提取并累加**，**不参与任何计费公式**。

### 3.2 数据来源

CodeBuddy 上游 SSE 最后一个 chunk 示例：

```json
{
  "choices": [...],
  "model": "...",
  "usage": {
    "prompt_tokens": 1234,
    "completion_tokens": 567,
    "cache_read_input_tokens": 0,
    "credit": 1.234                          // ← 关键字段
  }
}
```

`credit` 字段由 CodeBuddy 后端按模型 / token / cache 等内部定价策略算出，**对代理完全黑盒**。

### 3.3 提取路径（4 个协议入口）

所有路径都遵循同一个模式：累计响应流的最后一个 usage chunk → 调用 `recordUsage`：

| 协议 | 文件 | 提取位置 | recordUsage 时机 |
|---|---|---|---|
| OpenAI Chat Completions | [src/services/codebuddy/protocols/chat/completions.js#L88-L128](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/protocols/chat/completions.js#L88-L128) | 流式回调 + `aggregateStreamResponse` | 流结束 / 聚合完成 |
| Anthropic Messages | [src/services/codebuddy/protocols/anthropic/messages.js#L140-L215](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/protocols/anthropic/messages.js#L140-L215) | 同上 | 同上 |
| Responses HTTP | [src/services/codebuddy/protocols/responses/http.js#L140-L196](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/protocols/responses/http.js#L140-L196) | 同上 | 同上 |
| Responses Compact | [src/services/codebuddy/protocols/responses/compact.js#L65-L78](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/protocols/responses/compact.js#L65-L78) | 聚合完成后 | 同步一次 |
| Responses WebSocket | [src/services/codebuddy/protocols/responses/websocket.js#L120-L132](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/protocols/responses/websocket.js#L120-L132) | `onUsage` 回调 | 流结束时 |

```javascript

// 或

```

### 3.4 累计入口

所有 handler 都通过依赖注入拿到 `recordUsage`，实际实现统一走 [src/services/codebuddy/usage.js#L6-L35](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/usage.js#L6-L35)：

```javascript
export function recordCodebuddyUsage({
    tenantManager, tenantId,
    inputTokens, outputTokens, cacheHitTokens = 0,
    credit = 0,                              // ← 上游透传的 credit
    upstreamModel, clientModel
}) {
    if (!tenantId) return;
    tenantManager.incrementApiCallCount(tenantId, 'codebuddy');
    tenantManager.incrementTokenUsage(tenantId, 'codebuddy',
        inputTokens, outputTokens, cacheHitTokens);
    tenantManager.incrementCreditUsage(tenantId, 'codebuddy', credit);  // ← 累加 credit
    tenantManager.recordDailyUsage(tenantId, 'codebuddy',
        inputTokens, outputTokens, cacheHitTokens,
        credit,
        pickCodebuddyUsageModel(upstreamModel, clientModel));
}
```

四个动作：
1. `incrementApiCallCount` — 调用次数 +1
2. `incrementTokenUsage` — 累计 input / output / cache hit tokens
3. **`incrementCreditUsage` — 累计 credit（核心）**
4. `recordDailyUsage` — 写入日用量明细（按模型）

### 3.5 模型归属解析

[src/services/codebuddy/usage.js#L1-L4](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/usage.js#L1-L4)：

```javascript
export function pickCodebuddyUsageModel(upstreamModel, clientModel) {
    if (upstreamModel && !upstreamModel.startsWith('ep-')) return upstreamModel;
    return clientModel || upstreamModel;
}
```

如果上游返回的模型名以 `ep-` 开头（企业代理 ID），就回退用客户端传入的模型名作为统计维度，避免用量统计被企业代理 ID 污染。


```
        │
```

## 4. Relay 渠道 + 自建 LLM：能直接统计 credit 吗

### 4.1 简短结论

**不能直接统计，credit 永远为 0。**

### 4.2 Relay 的 recordUsage 接口没有 credit 参数

[src/services/relay/usage.js#L56-L84](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/relay/usage.js#L56-L84) 的接口签名：

```javascript

```

对比 codebuddy 渠道（[src/services/codebuddy/usage.js#L37-L60](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/usage.js#L37-L60)）：

```javascript
recordUsage(tenantId, inputTokens, outputTokens, cacheHitTokens, credit, upstreamModel, clientModel)
```

**Relay 的 `recordUsage` 完全没有 `credit` 参数**，且底层 [src/services/relay/usage.js#L29-L37](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/relay/usage.js#L29-L37) 直接把 credit 写死为 0：

```javascript
tenantManager.recordDailyUsage(
    tenantId,
    'relay',
    inputTokens,
    outputTokens,
    cacheHitTokens,
    0,                      // ← credit 永远是 0
    model
);
```

并且 relay 渠道**不会调用** `tenantManager.incrementCreditUsage`（[src/services/gateway/tenant-manager.js#L231-L238](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/gateway/tenant-manager.js#L231-L238)），所以 relay 渠道的 `credit` 字段永远不会被累加。

### 4.3 为什么 codebuddy 渠道能用

```jsonc
// CodeBuddy 上游 SSE 最后一个 chunk
{
  "usage": {
    "prompt_tokens": 1234,
    "completion_tokens": 567,
    "credit": 1.234        // ← 由 CodeBuddy 后端算好，代理只透传
  }
}
```

CodeBuddy 服务端**自带计费引擎**，把 `credit` 塞进 usage 字段，代理只需要在 [src/services/codebuddy/protocols/chat/completions.js#L98-L128](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/codebuddy/protocols/chat/completions.js#L98-L128) 之类的位置直接 `data.usage.credit || 0` 读出来就行。

### 4.4 自建 LLM 怎么办

自建 LLM 走 Relay 渠道，**你自己的上游响应里不会有 `credit` 字段**，OpenAI / Anthropic / vLLM 等标准协议的 `usage` 只有 tokens 信息。

**如果想统计"信用点 / 内部计费单位"**，需要自己加。最简单 3 种方式：

#### 方式 A：定义模型价格表，代理本地算

在 relay 路径上注册一个 `computeCredit(model, inputTokens, outputTokens)` 函数，然后改 [src/services/relay/usage.js](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/relay/usage.js) 的 `recordUsage`，让它接收 credit 并调用 `tenantManager.incrementCreditUsage`。

#### 方式 B：上游实现里塞 credit

在你的自建 LLM 代理层（vllm / ollama 之类的前置网关）合成符合 OpenAI 协议的响应，在 `usage` 字段里多塞一个 `credit`，代理侧就能像 codebuddy 渠道一样透传。

> 注意：relay 的 `recordUsage` 当前**没读取 `usage.credit` 字段**，所以方案 B 还得改 relay 代码。

#### 方式 C：直接用 dashboard 计费口径

如果只是要看用量，[src/services/gateway/tenant-manager.js](file:///c:/Users/sy100353/Trae/claude-api-proxy/src/services/gateway/tenant-manager.js) 已经有 `incrementApiCallCount` / `incrementTokenUsage` / `recordDailyUsage`，可以直接基于 token 数做内部计费。

### 4.5 限制总结

| 能力 | CodeBuddy 渠道 | Relay + 自建 LLM |
|---|---|---|
| 透传上游 credit | ✅ | ❌（上游一般没有这个字段） |
